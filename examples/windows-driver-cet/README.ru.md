# Драйвер ядра Windows с CET Shadow Stack

Минимальный драйвер ядра WDM, собранный с помощью NeverC, с включённым
Intel CET (Control-flow Enforcement Technology) Kernel Shadow Stack.
Кросс-компиляция с macOS / Linux.

## Сборка

```bash
cd examples/windows-driver-cet
make
```

Из автономной сборки NeverC:

```bash
make NEVERC=/path/to/neverc
```

Результат — `CetDriver.sys` (оптимизирован auto-LTO).
Сборка по умолчанию включает `-g` для отладки; **в релизных сборках следует убрать
`-g`**, чтобы удалить отладочные символы и уменьшить размер бинарного файла.

## Флаги, специфичные для CET

| Флаг | Уровень | Назначение |
|------|---------|-----------|
| `-fcf-protection=return` | Компилятор | Генерация кода, совместимого с Shadow Stack |
| `-Xlinker --cetcompat` | Линкер | Установка `IMAGE_DLL_CHARACTERISTICS_EX_CET_COMPAT` в PE |

## Ручная сборка (без Make)

```bash
neverc --target=x86_64-pc-windows-msvc \
  -g \
  -fcf-protection=return \
  -fms-kernel -fms-extensions -fms-compatibility \
  -D_AMD64_ -DNTDDI_VERSION=0x06010000 -D_WIN32_WINNT=0x0601 \
  -Wall -nostdlib -shared \
  -Xlinker --entry=DriverEntry \
  -Xlinker --subsystem=native \
  -Xlinker --nodefaultlib \
  -Xlinker --cetcompat \
  -lntoskrnl -lhal \
  -o CetDriver.sys driver.c
```

## Функциональность

- Создаёт объект устройства в `\Device\CetDriver`
- Создаёт символическую ссылку в `\DosDevices\CetDriver`
- Выполняет непрямые вызовы (указатель на функцию `ComputeFn`) для проверки совместимости с CET — Shadow Stack защищает адреса возврата этих вызовов
- Выводит сообщения о загрузке/выгрузке через `DbgPrint`

## Технические детали CET

CET имеет **два независимых механизма защиты**:

### 1. Shadow Stack — защита обратного края (RET)

Аппаратное обеспечение поддерживает второй стек (shadow stack), зеркалирующий операции CALL/RET.
**Специальные инструкции на входе в функцию не требуются** — полностью прозрачно:

```
┌─ CALL target ─────────────────────────────────┐
│                                                │
│  Обычный стек:  PUSH return_addr  (RSP)        │
│  Shadow stack:  PUSH return_addr  (SSP, HW)    │
│                                                │
└────────────────────────────────────────────────┘

┌─ RET ─────────────────────────────────────────┐
│                                                │
│  Обычный стек:  POP return_addr_A  (RSP)       │
│  Shadow stack:  POP return_addr_B  (SSP, HW)   │
│                                                │
│  Сравнение: return_addr_A == return_addr_B ?    │
│    ✓ совпадение → нормальный возврат            │
│    ✗ несовпадение → исключение #CP              │
│                                                │
└────────────────────────────────────────────────┘
```

### 2. Indirect Branch Tracking (IBT) — защита переднего края (непрямой CALL/JMP)

Требует инструкцию `ENDBR64` (`F3 0F 1E FA`, 4 байта) на каждой допустимой цели непрямого вызова/перехода. На CPU без CET `ENDBR64` является NOP.

### Выбор ядра Windows

| Защита | Механизм | Используется ядром Windows? |
|--------|---------|---------------------------|
| Обратный край (RET) | CET Shadow Stack | **Да** (KCET) |
| Передний край (непрямой CALL/JMP) | CET IBT (ENDBR) | **Нет** — используется CFG |

### Сравнение ассемблера: режимы `-fcf-protection`

Исходный код:

```c
unsigned long rotate13(unsigned long val) {
    return (val << 13) | (val >> 19);
}
```

#### `-fcf-protection=none` (без CET)

```asm
rotate13:
    mov  eax, ecx
    rol  eax, 13
    ret
```

#### `-fcf-protection=return` (только Shadow Stack — этот пример использует данный режим)

```asm
rotate13:
    mov  eax, ecx      ; идентично "none"!
    rol  eax, 13        ; Shadow Stack полностью прозрачен
    ret
```

#### `-fcf-protection=full` (Shadow Stack + IBT)

```asm
rotate13:
    endbr64             ; ← маркер IBT (F3 0F 1E FA)
    mov  eax, ecx
    rol  eax, 13
    ret
```

## Компилятор vs bin2bin: кто дружелюбен к CET?

CET проводит чёткую линию между **компиляторами на уровне исходного кода**
и **bin2bin-инструментами** (упаковщики, обфускаторы, хукеры, dump+rebuild).
Аппаратный Shadow Stack обеспечивает три правила, которые меняют всю
индустрию защиты / обфускации:

> 1. **Не модифицировать адреса возврата.**
> 2. **Не самопатчить код** (HVCI обеспечивает W^X на страницах кода).
> 3. **Искать сильные обфускационные преобразования**, уважающие 1 и 2.

### Может ли компилятор «зашифровать адреса возврата»?

**Нет.** Это распространённое заблуждение. Shadow Stack обеспечивается ЦП,
а не ОС, и невидим для пользовательского кода. Если XOR-зашифровать адрес
возврата на обычном стеке в эпилоге функции:

```c
void my_func() {
    // ... тело функции ...
    // эпилог пытается зашифровать адрес возврата:
    // XOR [rsp], 0xDEADBEEF
    // RET           <- железо сравнивает обычный стек vs теневой стек
                     //   они больше не совпадают -> исключение #CP -> BUGCHECK
}
```

Теневой стек по-прежнему хранит оригинальный адрес возврата. RET запускает
аппаратное сравнение; несовпадение вызывает `#CP` и bugcheck ядра. Компилятор
**не может** дотянуться до теневого стека:

- Пользовательский режим: ни одна инструкция не может писать в теневой стек
- Режим ядра: `WRSSQ` привилегированная, только `ntoskrnl` использует её

### CET-дружественные обфускации, которые компилятор МОЖЕТ делать

| Преобразование | Почему безопасно для CET |
|----------------|--------------------------|
| **Уплощение потока управления** | Switch-диспетчер использует прямые CALL/JMP; cases получают ENDBR64 при необходимости |
| **VM-виртуализация** | Обработчики соединены через косвенный JMP (с ENDBR64), не push+ret |
| **Шифрование строк / констант** | Чистое преобразование данных, нет влияния на поток управления |
| **MBA-выражения** | `x + y → (x ^ y) + 2*(x & y)` — только данные |
| **Непрозрачные предикаты** | Условные ветвления через прямые переходы |
| **Клонирование / инлайнинг функций** | Без изменения семантики стека вызовов |
| **Замена инструкций** | `MOV → XOR + ADD` — нет эффектов на стек |

### CET-враждебные паттерны (умирают под KCET)

| Паттерн | Почему ломается |
|---------|----------------|
| **Шифрование адреса возврата** | Несовпадение теневого стека → `#CP` |
| **PUSH addr; RET диспетчер** (классический стиль VMProtect / Themida) | То же — у теневого стека нет записи для `addr` |
| **Stack pivoting** (цепочки ROP-гаджетов) | Теневой стек не может следовать за pivot |
| **Самомодифицирующий код** | HVCI блокирует запись в исполняемые страницы |
| **Генерация кода во время выполнения** | То же — нарушение HVCI W^X |
| **Trampoline-основанные inline-хуки** | Модификация пролога функции запускает HVCI; даже обходя HVCI, теневой стек ломается на RET trampoline |

### Почему bin2bin-инструменты имеют структурный недостаток

Компилятор генерирует CET-корректный код из семантического IR. bin2bin-
инструмент должен **повторно открывать** семантику из скомпилированных байт:

1. **Неоднозначность границ инструкций** — x86 переменной длины. Добавление ENDBR64 (4 байта) по неверному смещению ломает всю RIP-относительную адресацию и релокации.
2. **Идентификация косвенных целей** — bin2bin не всегда может сказать, какие адреса в `.data` являются записями таблицы переходов vs сырые данные. Либо слишком сильно маркирует (раздутие кода, новые исходные точки для ROP-гаджетов), либо недостаточно (`#CP` во время выполнения).
3. **Опасность самоатестации** — Установка `IMAGE_DLL_CHARACTERISTICS_EX_CET_COMPAT` это обещание. Если bin2bin-вывод содержит любой CET-враждебный паттерн, драйвер прекрасно загрузится на не-CET машинах, но мгновенно BSOD на KCET хостах.
4. **Полнота CFG** — Компиляторы видят весь граф вызовов; bin2bin должен его выводить, а косвенные вызовы без точных целей вынуждают консервативное размещение ENDBR.

### Состояние индустрии

| Инструмент / класс | Состояние CET |
|--------------------|--------------|
| **NeverC / Clang / MSVC (компиляторы)** | Изначально CET-дружественны через `-fcf-protection` + флаг линкера |
| **OLLVM / Tigress / NeverC passes** | Преобразования на уровне IR → естественно CET-безопасны |
| **Microsoft Detours (4.0+)** | Обновлён для совместимости с CET |
| **VMProtect / Themida (старые)** | Push+RET диспетчер убивает драйвер на KCET хостах |
| **VMProtect / Themida (новые)** | Добавляют ENDBR-осведомлённые диспетчеры, смешанная поддержка |
| **Manual map / dump+rebuild загрузчики** | Должны реконструировать все маркеры ENDBR — подвержено ошибкам |

### Аспект игровой безопасности

Антициты-драйверы (EAC, BattlEye, FACEIT AC, Vanguard) поставляются с
установленным `--cetcompat`, поэтому они чисто работают на KCET-включённых
машинах. Чит-драйверы — обычно упакованные, хукнутые или внедрённые через
trampoline через bin2bin-инструментарий — с трудом остаются CET-совместимыми.
KCET + HVCI формируют **аппаратную стену "дружественную компиляторам,
враждебную bin2bin"**, которая асимметрично выгодна хорошо спроектированному
ПО безопасности по сравнению с кодом в стиле вредоносного ПО.

Это более глубокая причина, по которой Microsoft так сильно продвигает KCET
для ПО ядра: это делает легитимный код ядра проще для усиления, одновременно
делая ремесло атакующих постепенно более сложным.

## Активация KCET на целевой машине

```cmd
reg add HKLM\SYSTEM\CurrentControlSet\Control\DeviceGuard\Scenarios\HypervisorEnforcedCodeIntegrity /v Enabled /t REG_DWORD /d 1 /f
reg add HKLM\SYSTEM\CurrentControlSet\Control\DeviceGuard\Scenarios\KernelShadowStacks /v Enabled /t REG_DWORD /d 1 /f
```

Требуется перезагрузка. Проверьте через `msinfo32.exe`.

**Требования:** HVCI включён, Windows сборка 21389+, CPU с поддержкой CET (Intel Tiger Lake+ / AMD Zen 3+).

## Загрузка

```cmd
sc create CetDriver type= kernel binPath= C:\path\to\CetDriver.sys
sc start CetDriver
sc stop CetDriver
sc delete CetDriver
```

Включите тестовую подпись или используйте сертификат подписи кода для продакшена.

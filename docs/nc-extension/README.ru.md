# Расширение файла `.nc`

## Обзор

NeverC распознаёт `.nc` как своё нативное расширение исходных файлов. Когда компилятор обнаруживает входной файл `.nc`, он **автоматически включает** все расширения языка NeverC — дополнительные флаги не требуются.

## Автоматически включаемые функции

| Флаг | Эффект |
|------|--------|
| `-fneverc-types` | Псевдонимы целых чисел в стиле Rust (`u8`, `u16`, `u32`, `u64`, `i8`, `i16`, `i32`, `i64`, `usize`, `isize`) |
| `-fbuiltin-string` | Встроенный тип-значение `string` с автоматическим управлением памятью, синтаксисом dot-call и поддержкой UTF-8 |

## Использование

Просто назовите исходный файл с расширением `.nc`:

```bash
# Автоматически — -fbuiltin-string или -fneverc-types не нужны
neverc hello.nc -o hello

# Эквивалентно:
neverc -fneverc-types -fbuiltin-string hello.c -o hello
```

```c
// hello.nc
#include <stdio.h>

int main(void) {
    string greeting = "Привет, NeverC!";
    printf("%s (len=%zu)\n", greeting.c_str(), greeting.len);

    u32 x = 42;
    i64 y = -100;

    string msg = greeting + " x=%u, y=%lld".format(x, y);
    printf("%s\n", msg.c_str());
    return 0;
}
```

## Как это работает

Обнаружение происходит на двух уровнях конвейера компилятора:

### 1. Уровень Driver / Toolchain

Driver проверяет расширение каждого входного файла перед формированием вызова компилятора. Для файлов `.nc` флаги `-fneverc-types` и `-fbuiltin-string` безусловно добавляются в командную строку — пользователю не нужно передавать их вручную.

Для файлов `.c` эти флаги остаются опциональными: пользователь должен явно передать `-fneverc-types` и/или `-fbuiltin-string`.

### 2. Уровень CompilerInvocation

В качестве страховки фронтенд также проверяет расширения входных файлов при разборе вызова. Если какой-либо вход имеет расширение `.nc`, `LangOpts.NeverCTypes` и `LangOpts.BuiltinString` устанавливаются в `1`, гарантируя, что функции активны даже при обходе уровня driver (например, при прямом вызове `-cc1`).

## Совместимость

- Файлы `.nc` обрабатываются как исходный код C — язык по-прежнему C (по умолчанию C23), а не новый язык
- Все стандартные флаги C (`-std=c11`, `-O2`, `-g`, `-Wall` и т.д.) работают идентично
- `-fshellcode` естественно сочетается с `.nc`: режим shellcode уже включает `string`, а `.nc` обеспечивает активность `neverc-types`
- Кросс-компиляция (`-target aarch64-linux-gnu` и т.д.) работает так же
- Файлы `.c` не затрагиваются — они ведут себя точно так же, как раньше, если не передать флаги явно

## Когда использовать `.nc` vs `.c`

| Сценарий | Рекомендация |
|----------|-------------|
| Новый проект NeverC с `string` и типами в стиле Rust | Использовать `.nc` |
| Существующая кодовая база C, которая должна оставаться совместимой с другими компиляторами | Использовать `.c` + явные флаги |
| Проект shellcode | Оба варианта — `-fshellcode` включает `string` в любом случае |
| Смешанная кодовая база | `.nc` для файлов NeverC, `.c` для портируемого кода |
